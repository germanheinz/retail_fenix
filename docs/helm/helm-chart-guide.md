# Helm Chart Guide — Retail Fenix

## Overview

This guide documents how to create, structure, validate, and publish Helm charts for the Retail Fenix microservices project.

Each microservice has its own chart under `helm/`:

```
helm/
├── catalog/
├── cart/          (pending)
├── checkout/      (pending)
├── orders/        (pending)
└── ui/            (pending)
```

## Prerequisites

### Install Helm (Windows via Chocolatey)

```powershell
# Run PowerShell as Administrator
choco install kubernetes-helm -y
```

## Creating a New Chart

```bash
cd retail_fenix/helm

# Creates the full chart scaffold
helm create <service-name>

# Example
helm create catalog
```

This generates:

```
catalog/
├── .helmignore
├── Chart.yaml
├── values.yaml
└── templates/
    ├── _helpers.tpl
    ├── deployment.yaml
    ├── service.yaml
    ├── hpa.yaml
    ├── ingress.yaml
    ├── serviceaccount.yaml
    ├── NOTES.txt
    └── tests/
        └── test-connection.yaml
```

## Chart Structure for Retail Fenix

After cleanup, each chart keeps only what is needed:

```
catalog/
├── .helmignore
├── Chart.yaml
├── values.yaml
└── templates/
    ├── _helpers.tpl          # reusable template functions
    ├── configmap.yaml        # app environment variables
    ├── deployment.yaml       # pod spec
    ├── service.yaml          # ClusterIP service
    └── tests/
        └── test-connection.yaml
```

Files to delete from the generated scaffold (not needed):
- `hpa.yaml`
- `ingress.yaml`
- `serviceaccount.yaml`
- `httproute.yaml`
- `NOTES.txt`

## Key Files

### Chart.yaml

Defines chart metadata. Always update `appVersion` to match the Docker image tag.

```yaml
apiVersion: v2
name: catalog
description: Helm chart for Retail Fenix catalog service
type: application
version: 1.0.0       # bump this when chart changes
appVersion: "1.0.0"  # matches Docker image tag
```

### values.yaml

Defines all configurable values. The templates reference these via `.Values.*`.

```yaml
replicaCount: 1

image:
  repository: public.ecr.aws/<alias>/retail-fenix/catalog
  tag: "1.0.0"
  pullPolicy: IfNotPresent

service:
  type: ClusterIP
  port: 8080

serviceAccount:
  create: false
  name: "catalog-mysql-sa"

podSecurityContext:
  runAsNonRoot: true

securityContext:
  runAsUser: 1000
  runAsNonRoot: true
  readOnlyRootFilesystem: true
  allowPrivilegeEscalation: false
  capabilities:
    drop:
      - ALL

resources:
  requests:
    cpu: "250m"
    memory: "128Mi"
  limits:
    cpu: "500m"
    memory: "256Mi"

livenessProbe:
  httpGet:
    path: /health
    port: 8080
  initialDelaySeconds: 15
  periodSeconds: 10
  failureThreshold: 3

readinessProbe:
  httpGet:
    path: /health
    port: 8080
  initialDelaySeconds: 15
  periodSeconds: 10
  failureThreshold: 3

volumeMounts:
  - name: aws-secrets
    mountPath: /mnt/secrets-store
    readOnly: true

volumes:
  - name: aws-secrets
    csi:
      driver: secrets-store.csi.k8s.io
      readOnly: true
      volumeAttributes:
        secretProviderClass: "catalog-db-secrets"

autoscaling:
  enabled: false

app:
  persistence:
    provider: mysql
    endpoint: ""
    database: "catalogdb"
    secret:
      name: catalog-db
```

### templates/configmap.yaml

Must be created manually — not generated by `helm create`.

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: {{ include "catalog.fullname" . }}
  labels:
    {{- include "catalog.labels" . | nindent 4 }}
data:
  RETAIL_CATALOG_PERSISTENCE_PROVIDER: {{ .Values.app.persistence.provider | quote }}
  RETAIL_CATALOG_PERSISTENCE_ENDPOINT: {{ .Values.app.persistence.endpoint | quote }}
  RETAIL_CATALOG_PERSISTENCE_DB_NAME: {{ .Values.app.persistence.database | quote }}
  RETAIL_CATALOG_PERSISTENCE_CONNECT_TIMEOUT: "5"
```

## Useful Commands

### Validate

```bash
# Check chart for errors
helm lint catalog

# Render YAML without installing (dry run)
helm template catalog catalog

# Dry run against a live cluster
helm install catalog ./catalog --dry-run --debug
```

### Install / Upgrade

```bash
# Install
helm install catalog ./catalog

# Install with custom values file
helm install catalog ./catalog -f custom-values.yaml

# Upgrade (or install if not exists)
helm upgrade --install catalog ./catalog

# Upgrade with a specific image tag
helm upgrade --install catalog ./catalog --set image.tag=1.1.0
```

### Inspect a release

```bash
# List all releases
helm list

# Show release status and resources
helm status catalog --show-resources

# Show computed values
helm get values catalog

# Show rendered manifests of a deployed release
helm get manifest catalog
```

### Rollback

```bash
# Rollback to previous revision
helm rollback catalog

# Rollback to a specific revision
helm rollback catalog 2
```

### Uninstall

```bash
helm uninstall catalog
```

## Package and Push to ECR Public

### Step 1 — Login to ECR Public

```bash
# ECR Public always uses us-east-1
aws ecr-public get-login-password --region us-east-1 \
  | helm registry login --username AWS --password-stdin public.ecr.aws
```

### Step 2 — Get your ECR Public alias

```bash
aws ecr-public describe-registries --region us-east-1 \
  --query 'registries[0].aliases[0].name' --output text
```

### Step 3 — Create the repository

```bash
aws ecr-public create-repository \
  --repository-name retail-fenix/catalog \
  --region us-east-1
```

### Step 4 — Package the chart

```bash
cd retail_fenix/helm
helm package catalog
# generates: catalog-1.0.0.tgz
```

### Step 5 — Push to ECR Public

```bash
helm push catalog-1.0.0.tgz oci://public.ecr.aws/<your-alias>
```

### Step 6 — Verify

```bash
aws ecr-public describe-image-tags \
  --repository-name retail-fenix/catalog \
  --region us-east-1
```

### Install directly from ECR Public

```bash
helm install catalog \
  oci://public.ecr.aws/<your-alias>/retail-fenix/catalog \
  --version 1.0.0
```

## Push to ECR Private

### Step 1 — Set variables

```bash
REGION=us-east-1
ACCOUNT_ID=$(aws sts get-caller-identity --query Account --output text)
REGISTRY="${ACCOUNT_ID}.dkr.ecr.${REGION}.amazonaws.com"
```

### Step 2 — Login

```bash
aws ecr get-login-password --region "$REGION" \
  | helm registry login -u AWS --password-stdin "$REGISTRY"
```

### Step 3 — Create repository

```bash
aws ecr create-repository \
  --repository-name retail-fenix/catalog \
  --region "$REGION"
```

### Step 4 — Package and push

```bash
helm package catalog
helm push catalog-1.0.0.tgz oci://"$REGISTRY"
```

### Step 5 — Install from ECR Private

```bash
helm install catalog \
  oci://"$REGISTRY"/retail-fenix/catalog \
  --version 1.0.0
```

## Adding a New Microservice Chart

Repeat the same process for each service:

```bash
cd retail_fenix/helm
helm create cart
# then edit Chart.yaml, values.yaml, templates/
```

Key differences per service:
- `image.repository` — points to that service's Docker image
- `service.port` — the port the service listens on
- `configmap.yaml` — environment variables specific to that service
- `app.persistence` — only needed for services that connect to a DB

## Troubleshooting

### ECR Login fails on Windows — "the stub received bad data"

**Problema:** Al intentar hacer login a ECR Public en Windows (Git Bash o PowerShell) usando el pipe `|`, aparece el error:

```
the stub received bad data
```

O al usar `--password-stdin`:

```
failed to perform exits on destination HEAD
```

**Causa:** El AWS CLI v2 en Windows usa un wrapper `.exe` que no es compatible con el pipe estándar de PowerShell ni de Git Bash.

**Solución:** Usar `docker login` en lugar de `helm registry login`. Helm usa el mismo credential store que Docker, por lo que si Docker está logueado, Helm también puede hacer push.

```powershell
# Paso 1 — Capturar el token en una variable (no usar pipe directo)
$token = & "C:\Program Files\Amazon\AWSCLIV2\aws.exe" ecr-public get-login-password --region us-east-1

# Paso 2 — Login con Docker en lugar de Helm
docker login public.ecr.aws --username AWS --password $token

# Paso 3 — Push con Helm (usa las credenciales de Docker automáticamente)
helm push catalog-1.0.0.tgz oci://public.ecr.aws/i5b4r2o0/retail-fenix
```

> Usar `& "C:\full\path\aws.exe"` en lugar de solo `aws` resuelve el problema de que PowerShell no encuentre el comando en el PATH.

**Por qué funciona:** Helm delega la autenticación al credential store del sistema (el mismo que usa Docker). Al hacer `docker login`, las credenciales quedan guardadas y Helm las usa automáticamente al hacer `helm push`.

---

### ECR Push devuelve 403 Forbidden

**Causa:** El token de ECR expira rápidamente. Si pasó tiempo entre el login y el push, el token ya no es válido.

**Solución:** Hacer el login y el push en la misma sesión sin pausa.
